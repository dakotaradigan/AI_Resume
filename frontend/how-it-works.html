UNDER THE HOOD

How this site works

A production AI application, built and measured like one — every claim below is backed by code and committed eval results in the repo.

Real streaming

Hand-rolled server-sent events — session, status, token deltas, done — no fake typing animations.

Hybrid retrieval

Dense vectors + BM25 fused with reciprocal rank fusion over a 25-chunk corpus of resume and project docs.

Measured

Owner-approved golden set: hit-rate@4 0.886, MRR 0.838. Three LLM judges score production answers.

Hardened

Strict CSP, server-minted HttpOnly identity, prompt-injection firewall, atomic budgets, hashed analytics.

Architecture

One FastAPI service serves the static frontend and the API. Guardrails (rate limits, quotas, validation) run before any model call; retrieval and model routing run concurrently; the answer streams back over SSE.

Streaming & model routing

Every chat is a real server-sent-event stream: the browser shows retrieval status, then tokens as Claude produces them. What you see in the status steps is what actually happened — a cached answer says "answered from cache," it doesn't fake a search.

A small Haiku classifier (4 output tokens, 2-second timeout) routes each question: simple factual lookups go to Sonnet, synthesis and comparisons go to Opus, and any router failure fails safe to the most capable model. Routing runs concurrently with retrieval, so it adds no time to first token.

event: status   {"stage": "rag_search", "state": "done", "sources": [...]}
event: status   {"stage": "routing", "state": "done", "model": "sonnet"}
event: delta    {"text": "Dakota is PCAP-certified and uses Python in"}
event: done     {"reply": "...", "sources": [...], "followups": [...]}

Retrieval

Questions search a 25-chunk corpus built from the structured resume and full project write-ups. Retrieval is hybrid: OpenAI text-embedding-3-small vectors in Qdrant plus an in-process BM25 keyword index, fused with reciprocal rank fusion. Vector search catches paraphrases; BM25 catches exact terms like certification names that embeddings blur.

If retrieval is unavailable, the site degrades gracefully to full-resume context — it never fails a visitor because a vector database was down.

Evals

Retrieval quality is measured against a 35-query golden dataset, reviewed and approved by Dakota before any number was recorded. Baseline (2026-07-19): hit-rate@4 0.886 · recall@4 0.886 · MRR 0.838, clearing the 0.85 quality gate.

Answer quality is scored by three LLM judges — groundedness, conciseness, and redirect handling — plus thumbs feedback from real visitors. Prompt changes are re-evaluated before they ship.

Security

The content-security policy allows scripts from this origin only — no inline scripts, no CDNs. Quotas and unlocks are keyed to a server-minted HttpOnly cookie, not anything a client can forge. Pasted job descriptions are sanitized and firewalled as untrusted data, model spend is capped by atomic reserve-and-release budgets, and analytics store only HMAC-hashed identifiers.

Connect your own AI

The resume is machine-readable too: /llms.txt is rendered live from the same source of truth, and an MCP endpoint exposes one data-only tool, get_resume. Point your Claude at it:

claude mcp add --transport http resume https://www.dakotaradigan.io/mcp

Why no LangChain?

The direct SDK calls here are about thirty lines, including retries and timeouts. Frameworks earn their keep when you're swapping providers or composing many chains — neither applies to this site. The effort went into measured retrieval, validated evals, and security hardening instead. Every abstraction here is one I can defend line by line.
